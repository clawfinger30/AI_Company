## 1. 概要

`00_ai_company` は、`C:\App_Make` 配下のアプリ開発を支援する AI 社員の規則、役割、アプリ別ルール、仕様変更履歴を管理するファイルベースの運用基盤である。

現行実体は `00_ai_company/` 配下の Markdown 文書で構成される。常駐プロセス、CLI、Web UI、データベース、外部 API サーバーは含まれない。AI 社員が各文書を所定の順番で読み、会話と実装コードを調査し、仕様変更案やレポートをファイルとして保存することで動作する。

本仕様書では、共通基盤名を `00_ai_company`、構成員を「AI 社員」と表記する。特定アプリ名は `registry/apps.md` の正式名で扱う。

## 2. 目的

- AI 作業時に、会話、仕様書、実装コード、変更履歴の対応関係を維持する。
- 複数アプリに共通する AI 作業規則を一元管理する。
- AI 社員の職種ごとの責務と、アプリ固有の注意事項を分離する。
- 仕様の追加、変更、廃止候補、コードとの差分、未確定事項を追跡可能な形式で残す。
- 危険な変更や削除を承認フローに載せ、未承認の直接反映を防ぐ。
- バージョン、Git 状態、検証結果、生成物を作業記録に残す。
- コード作成時は、機械的な完了判定、複数解釈の確認、ついで改善禁止、検証報告、同一エラー2回制限、完了前レビュー、進捗・確信度・残リスク報告を必須とする。

## 3. 対象範囲

### 3.1 対象

- `00_ai_company/company_rules/` の全社共通規則
- `00_ai_company/ai_employees/` の職種定義
- `00_ai_company/app_departments/` のアプリ部署規則

- `00_ai_company/runtime/` の運用記録

### 3.2 対象外

- 各管理対象アプリ本体の詳細機能
- AI モデルやチャットクライアントの実装
- OS レベルのファイル監視
- Windows タスクスケジューラーのジョブ
- Git hook の実装
- 承認画面やワークフローエンジン
- ユーザー認証、権限管理、課金

## 5. 規則の適用順序

AI 社員は、対象アプリに対する作業前に次の順序で情報を読む。

1. `company_rules/employee-handbook.md`
2. `company_rules/approval-policy.md`
3. `company_rules/history-policy.md`
4. `company_rules/change-tracking-policy.md`
5. `company_rules/file-operation-policy.md`
6. `ai_employees/{職種}/role.md`
7. `app_departments/{アプリ}/app-rules.md`
8. `app_departments/{アプリ}/employees/{職種}.md`
9. 対象アプリの既存仕様書
10. 対象アプリの実装コード

後段の文書は対象アプリや職種に固有の条件を追加する。ただし、全社共通の禁止事項や承認要件を解除する規則は現行文書に定義されていない。

## 6. AI 社員仕様

### 6.1 登録済み職種

| AI 社員 | 状態 | 主な責務 |
| --- | --- | --- |
| 仕様書メモAI | 運用開始 | 会話・仕様・コードの差分抽出、変更案・レポート・会話ログの作成 |
| コードレビューAI | 役割定義のみ | バグ、仕様逸脱、テスト不足、運用リスクの指摘 |
| テスト作成AI | 役割定義のみ | テスト観点の抽出、テスト提案、テストコード作成 |
| リリース管理AI | 役割定義のみ | リリース前確認、変更履歴整理、配布前レポート作成 |

仕様書メモAI以外の職種には、現時点で詳細ワークフロー、出力スキーマ、アプリ専属ルールが整備されていない。

### 6.2 仕様書メモAI

仕様書メモAIは次の情報を区別して扱う。

- ユーザーが明確に指示した仕様
- AI が提案しただけの仕様
- 実装コードから読み取れる仕様
- 仕様書にあるがコード確認できていない仕様
- コードにあるが仕様書にない仕様

差分は次の区分で分類する。

| 区分 | 意味 |
| --- | --- |
| `add` | 新規仕様 |
| `change` | 既存仕様の変更 |
| `deprecate` | 廃止候補 |
| `remove` / `remove_candidate` | 削除候補。直接削除は禁止 |
| `question` | 未確定事項 |
| `code_gap` | コードと仕様書の差分 |

### 6.3 自動記録条件

AI 社員が次の作業を扱った場合、仕様書メモAIは明示依頼がなくても記録対象とする。

- 新しい仕様、仕様変更、仕様廃止、運用判断の指示
- 画面、API、DB、設定、外部連携、スケジュール、認証、通知、投稿処理等の挙動変更
- コードと仕様書の差分、未記載仕様、未確定事項の発見
- リリース、ビルド、バージョン更新、Git コミットに関係する追跡情報

次の場合は自動記録対象外とする。

- 仕様、コード、運用履歴に影響しない質問回答
- 新しい判断や差分を伴わないファイル閲覧
- 仕様として残す根拠がない一時的な調査ログ

自動記録の範囲は、会話ログ、仕様変更案、差分レポートの作成までである。既存仕様書本文への自動反映は行わない。

## 8. 標準作業フロー

仕様書メモAIの標準処理は次のとおりである。

1. 全社共通ルールを読む。
2. 仕様書メモAIの職種定義を読む。
3. 対象アプリ部署ルールを読む。
4. 対象アプリ専属ルールを読む。
5. 会話ログを確認する。
6. 既存仕様書を確認する。
7. 関連コードを確認する。
8. 仕様差分を分類する。
9. バージョン、Git状態、対象ファイル、検証方法を確認する。
10. 仕様変更案またはレポートを作成する。
11. 承認が必要な項目を分離する。
12. 変更追跡ポリシーの必須項目を確認する。

## 9. 仕様変更管理

### 9.1 状態

```text
draft
  └─ pending
       ├─ approved
       │    └─ applied
       └─ rejected
```

| 状態 | 保存先または意味 |
| --- | --- |
| `draft` | 下書き |
| `pending` | `runtime/spec_changes/pending/`。承認待ち |
| `approved` | `runtime/spec_changes/approved/`。承認済み、未反映 |
| `applied` | `runtime/spec_changes/applied/`。仕様書本文へ反映済み |
| `rejected` | `runtime/spec_changes/rejected/`。却下理由を保持 |
| `deprecated` | 廃止予定または廃止済み |

状態フォルダ間の移動を自動化するプログラムは存在しない。AI 社員または利用者が規則に従ってファイルを作成・移動する運用である。

### 9.2 仕様変更案の保存先

```text
runtime/spec_changes/pending/{app}/{YYYYMMDD_HHmmss}_{summary}.md
```

仕様変更案には、変更ID、追跡ID、対象アプリ、変更種別、根拠会話、確信度、承認要否、バージョン、Git状態、変更ファイル、未確定事項を記録する。

## 10. 承認仕様

### 10.1 承認なしで作成可能

- 新しい仕様変更案
- 会話ログ
- 差分レポート
- 未確定事項リスト
- 既存仕様書への参照リンク
- 既存仕様書本文を変更しない運用ルール文書

### 10.2 承認後に反映

- 既存仕様書本文への大きな変更
- 仕様の廃止または削除
- DB スキーマ、認証、外部 API、課金、法務に関係する変更
- 複数アプリに影響する共通仕様変更
- 各アプリ部署ルールで承認対象に指定された変更

### 10.3 判断保留

次の項目は確定仕様にせず、`確認待ち` として残す。

- 曖昧なユーザー発言
- AI が推測した仕様
- 仕様書とコードのどちらを正とするか判断できない差分
- 実装確認ができていない仕様

承認を記録・検証する専用システムは存在しない。承認者、承認日時、根拠会話を Markdown のメタデータまたは本文に記録する。

## 11. 削除・廃止仕様

AI 社員は仕様やファイルを直接削除しない。削除が必要な場合は次の順で扱う。

1. `deprecated` または `archive_candidate` として候補化する。
2. 代替仕様がある場合は `replaced_by` を記録する。
3. 廃止理由、根拠会話ID、確認者を記録する。
4. 承認後に archive へ移動する。

削除候補の標準記録形式は次のとおりである。

```yaml
status: deprecated
deprecated_reason:
replaced_by:
source_conversation_id:
```

archive の具体的な保存先と保持期間は現行仕様で定義されていない。

### 12.1 文書形式

- 規則、仕様書、変更案、レポート: Markdown

### 12.3 共通追跡メタデータ

仕様変更案、仕様書、作業レポート、リリースレポートでは、可能な限り次の情報を保持する。

- 追跡ID
- 対象アプリとパス
- AI 社員名
- 作業種別と状態
- 作成日時、更新日時、追加日時、変更日時、廃止日時
- 根拠会話IDと依頼要約
- 変更前後のアプリバージョンと版数ファイル
- Git リポジトリ、ブランチ、変更前後コミット、状態、タグ、リモート
- 変更ファイル、変更種別、理由、根拠
- 検証コマンド、結果、補足
- 生成物のパス、作成日時、補足
- 承認要否、理由、承認者、承認日時
- 未コミット差分、Git管理外ファイル、残課題

取得できない値は空欄にせず、取得できない理由を記録する。

### 12.4 会話ログ

保存先:

```text
runtime/conversation_logs/{YYYY-MM-DD}.jsonl
```

レコード例:

```json
{

  "turn_id": "turn-0001",
  "timestamp": "2026-07-09T15:50:34+09:00",
  "app": "00_ai_company",
  "speaker": "user",
  "message": "00_ai_companyの内容を汎用AI社員基盤として整理し、重点管理アプリを管理できるようにする",
  "summary": "汎用AI社員基盤化と重点管理アプリ追加の依頼",
  "tags": ["spec", "management", "priority-apps"],
  "source": "chat"

```

必須項目は `conversation_id`、`turn_id`、`timestamp`、`speaker`、`message` である。`app`、`summary`、`tags`、`source`、`related_files`、`confidence` は任意である。

## 16. 非機能仕様

### 16.1 追跡可能性

- 仕様、コード、会話、変更理由を相互に追跡できる情報を残す。
- Git 管理下では作業前後のブランチ、コミット、状態を記録する。
- Git 管理外では管理外である理由を明記する。
- テストやビルドを実施しない場合も `skipped` と理由を残す。

### 16.2 安全性

- 根拠のない仕様を確定扱いにしない。
- 実装未確認の状態で「コードと一致」と記載しない。
- 既存仕様やファイルを直接削除しない。
- アプリ間で仕様を混在させない。
- 会話中の仮案をユーザーの確定指示として扱わない。
- 認証、課金、DB、外部API、法務関連の変更は承認対象とする。
- コード作成、コード修正、ビルド、リリース作業では、完了条件をコマンド、差分、テスト結果、生成物、画面確認などで機械的に判定できる形にする。
- 要件に複数の妥当な解釈がある場合は、勝手に一つへ確定せず、採用した解釈と理由を明記する。仕様、データ破壊、外部連携、課金、認証、公開動作に影響する場合は確認待ちにする。
- 依頼目的に直接必要な変更以外のついで改善を行わない。追加改善は実施せず、提案または残課題として記録する。
- 完了報告では「動いた」ではなく、実行した検証コマンド、確認対象、結果、未検証理由、残リスクを報告する。
- 同一原因または同一症状に対する修正試行は2回までとし、2回失敗した場合は原因仮説、試行内容、残る選択肢、必要な確認を整理して報告する。
- 完了前に初見レビューを行い、仕様逸脱、不要変更、テスト不足、破壊的変更、セキュリティ/運用リスク、未承認事項を確認する。
- 作業中または完了時は、必要に応じて `進捗`、`確信度`、`根拠/残リスク` の3点で報告する。

### 16.3 保守性

- 共通規則、職種規則、アプリ固有規則を分離する。
- 新規アプリと新規職種は、既存のディレクトリ構造とテンプレートを使って追加する。
- 運用記録を仕様書本文から分離して `runtime/` に保存する。

## 17. 画面仕様

`00_ai_company` 自体には画面、Web UI、デスクトップ UI、コマンドライン UI は存在しない。利用者はファイルシステムと AI 会話を通じて操作する。

## 18. 外部連携

`00_ai_company` 自体に外部 API 接続処理は存在しない。

管理対象アプリには X、Pinterest、Instagram Graph API、楽天ROOM、楽天アフィリエイト、Nextcloud、Google カレンダー等の連携があるが、それらの詳細仕様は各アプリの仕様書で管理する。

Git、Windows タスクスケジューラー、ファイル監視は運用上の連携候補として定義されているが、`00_ai_company` 内に実行コードはない。

## 20. 運用上の完了条件

仕様影響がある作業は、次のいずれかを満たした時点で記録上の完了とする。

- 会話ログ、仕様変更案、差分レポートのいずれかを作成した。
- 承認待ちの仕様変更案を `pending` に作成した。
- 仕様とコードの差分レポートを作成した。
- 変更不要と判断した根拠をレポートに残した。
- 人間による確認が必要な未確定事項を明示した。

コード変更、ビルド、リリースを伴う場合は、変更ファイル、版数、Git状態、検証、生成物、残課題も記録する。

コード作成を伴う作業は、以下も満たした時点で完了扱いにする。

- 完了条件を機械的に確認できる形で満たした。
- 複数解釈がある要件について、確認済みまたは採用理由を明示した。
- 依頼外のついで改善を含めていない。
- 実行した検証と結果を記録した。
- 同一エラーへの修正試行が2回を超えていない。
- 完了前の初見レビューを実施した。
- 進捗、確信度、根拠/残リスクを必要に応じて報告した。
