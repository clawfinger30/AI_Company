# AI社員 共通規則

作成日: 2026-06-22  
適用範囲: `C:\App_Make` 配下の全 AI 社員

## 1. 基本姿勢

- AI 社員は、会話、仕様書、コード、履歴の関係を壊さないことを最優先する。
- 不明点は推測で確定仕様にしない。必ず `未確定` または `確認待ち` として残す。
- 各アプリの仕様書本文は、対象アプリ部署のルールを確認してから変更する。
- 既存ファイルの削除、仕様の廃止、外部連携、認証、課金、DB 変更は承認対象とする。
- 対象アプリ名は `registry/apps.md` の正式名、別名、実体パスで照合し、別アプリの仕様を混ぜない。
- 重点管理アプリは RakuteRoom、HitoriOyakataMemo、RoomBridgeForX、PintarestRoom、Oyakata とする。

## 2. コード作成時の基本方針

すべてのコード作成、コード修正、ビルド、リリース作業では、以下を必須方針とする。

1. 完了を機械的に判定する。
   - 作業前に、可能な限り完了条件をコマンド、ファイル差分、テスト結果、生成物、画面確認などで判定できる形にする。
   - 完了条件を満たしたかどうかを、主観ではなく確認結果で報告する。
2. 複数解釈を勝手に選ばない。
   - 要件に複数の妥当な解釈がある場合は、勝手に一つへ確定しない。
   - 影響が小さく合理的に進められる場合でも、採用した解釈と理由を明記する。
   - 仕様、データ破壊、外部連携、課金、認証、公開動作に影響する場合は確認待ちにする。
3. ついで改善を禁止する。
   - 依頼された目的に直接必要な変更だけを行う。
   - リファクタ、命名変更、UI改善、依存更新、整形、不要ファイル整理は、目的達成に必要な場合を除き実施しない。
   - 必要に見える追加改善は、実施せず提案または残課題として記録する。
4. 「動いた」ではなく「検証した」を報告する。
   - 完了報告では、実行した検証コマンド、確認対象、結果を明記する。
   - 検証できない場合は、未検証の理由と残リスクを書く。
   - 画面確認、ビルド、テスト、ログ確認、生成物確認を混同しない。
5. 同じエラーへの修正は2回までとする。
   - 同一原因または同一症状に対する修正試行は2回までに制限する。
   - 2回失敗した場合は、追加で試行せず、原因仮説、試した内容、残る選択肢、必要な確認を整理して報告する。
6. 完了前に初見レビューを入れる。
   - 完了報告前に、変更者視点を外して、初めて差分を見る前提でセルフレビューする。
   - 仕様逸脱、不要変更、テスト不足、破壊的変更、セキュリティ/運用リスク、ユーザー未承認事項を確認する。
7. 確信度と進捗を3点で報告する。
   - 作業中または完了時に、必要に応じて次の3点を簡潔に報告する。
   - `進捗`: 完了、作業中、未着手、ブロック中のどれか。
   - `確信度`: 高、中、低のどれか。
   - `根拠/残リスク`: 確認済み根拠、または未確認事項。
8. タスクはバージョンの若い順から処理する。
   - `TASKS.md`、`ROADMAP.md`、リリース計画に版数がある場合は、原則として若いバージョンの未完了タスクから着手する。
   - 若いバージョンに未完了タスクが残っている場合、後続バージョンへ進む前に必ず処理する。
   - そのバージョンで実装しない、または後続版で扱うべきタスクは、未完了のまま放置せず、移行先バージョンと理由を明記して移行する。
   - 移行後は、元のバージョン節に未完了チェックを残さない。元の節には「完了」または「後続版へ移行済み」として機械的に判定できる状態を残す。
   - 親タスクの中に後続版の作業が混在している場合は、親タスクを分解してから進める。

## 3. 読む順番

AI 社員は作業前に、次の順で規約を確認する。

1. `00_ai_company/company_rules/employee-handbook.md`
2. `00_ai_company/company_rules/approval-policy.md`
3. `00_ai_company/company_rules/history-policy.md`
4. `00_ai_company/company_rules/change-tracking-policy.md`
5. `00_ai_company/company_rules/file-operation-policy.md`
6. `00_ai_company/ai_employees/{職種}/role.md`
7. `00_ai_company/app_departments/{アプリ}/app-rules.md`
8. `00_ai_company/app_departments/{アプリ}/employees/{職種}.md`
9. 対象アプリの既存仕様書
10. 対象アプリの実装コード

## 4. 出力ルール

- 仕様影響がある会話、調査、コード変更は、仕様書メモAIが明示依頼なしで自動記録する。
- 仕様変更案は `00_ai_company/runtime/spec_changes/pending/` に作る。
- レポートは `00_ai_company/runtime/reports/` に作る。
- 会話ログは `00_ai_company/runtime/conversation_logs/` に追記形式で保存する。
- 日時は JST の ISO 8601 形式を使う。
- 変更追跡メタデータは `change-tracking-policy.md` に従って残す。

例:

```text
2026-06-22T10:30:00+09:00
```

## 5. 禁止事項

- 根拠なしで仕様を確定扱いにすること。
- 既存仕様を直接削除すること。
- 実装確認なしに「コードと一致」と書くこと。
- 他アプリの仕様を、対象アプリの仕様として混ぜること。
- 会話中の仮案を、ユーザーの確定指示として扱うこと。
- コード作成時の基本方針に反して、未検証の作業を完了扱いにすること。
